Windows account policy sets the rules; this lesson is about auditing the
accounts those rules apply to. Almost everything here is PowerShell, and
almost everything here starts from the README's authorized-user and
authorized-admin lists.

## Listing users and group membership

```
Get-LocalUser | Select Name,Enabled,LastLogon
Get-LocalGroupMember -Group "Administrators"
```

The first line is your full account inventory — cross-reference every name
against the README's authorized-standard-user list. The second is your
admin-membership check — cross-reference against the authorized-administrator
list specifically. These are two different lists; a user can be authorized as
a standard user but *not* authorized as an admin, in which case the fix is to
demote them from Administrators, not delete the account.

```
Remove-LocalUser -Name "baduser"
Remove-LocalGroupMember -Group "Administrators" -Member "authorizeduser"
```

Delete an account only if it's absent from both lists. If it's a legitimate
user who simply shouldn't be an admin, remove them from the group instead —
deleting an authorized account is a penalty.

## Weak and blank passwords

There's no single PowerShell one-liner that tells you a password is "weak" —
Windows doesn't expose plaintext passwords for inspection. In practice this
category means: set a strong password on every authorized account, and don't
leave any account with a blank password.

```
$p = Read-Host -AsSecureString
Set-LocalUser -Name "alice" -Password $p
```

Keep the password your team settles on written in the change log (per the
safe-change-discipline lesson) so anyone on the team can log back in.

## Password-never-expires

This flag is a common miss because it silently undermines the age policy
from the previous lesson — an account can have "password never expires" set
even after you've configured `net accounts /maxpwage:90`, and that one
account simply won't be subject to it.

```
Get-LocalUser | Where-Object { $_.PasswordExpires -eq $null -and $_.Enabled }
Set-LocalUser -Name "alice" -PasswordNeverExpires $false
```

`PasswordExpires` returns `$null` specifically when the never-expires flag is
set on an enabled account — that's the signal this query is looking for.

## Guest and the built-in Administrator

Two accounts exist on every fresh Windows install and are scored by default
unless the README says otherwise:

```
Get-LocalUser -Name "Guest","Administrator" | Select Name,Enabled
Disable-LocalUser -Name "Guest"
Disable-LocalUser -Name "Administrator"
```

Neither account can be deleted through normal means — Guest and the built-in
Administrator are baked into Windows — so the fix is always **disable**, not
delete. If the README explicitly requires either one active (uncommon but it
happens), leave it enabled and move on; don't disable an account the README
needs.

## Order of operations

A practical sequence that avoids locking yourself out mid-audit:

1. Pull the full user list and admin-group list first, before changing
   anything.
2. Cross-reference both against the README's two lists.
3. Fix passwords/never-expires on accounts you're keeping.
4. Remove unauthorized accounts and demote unauthorized admins.
5. Handle Guest/built-in Administrator last, since disabling the wrong
   account here is the one mistake in this list that can actually lock a
   teammate out.
