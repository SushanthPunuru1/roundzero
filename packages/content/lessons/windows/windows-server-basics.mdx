Everything so far in this pillar targets a standalone Windows workstation —
the image type most CyberPatriot Windows rounds actually use. Server images
show up in later rounds and bring one structural difference worth
understanding even briefly: on a domain-joined network, policy and
administration live at the *domain* level, not just the local machine. This
lesson is a foundation, not a full server-hardening pass — hands-on Server
work (like everything else in this pillar) happens on a real box.

## Role minimization

Windows Server runs almost nothing by default until you explicitly add
"roles" (DNS Server, Active Directory Domain Services, IIS, DHCP Server) and
"features" (smaller add-ons). That's the opposite failure mode from a
workstation image, where services tend to arrive pre-installed — on Server,
someone chose to install each role, and every one installed-but-not-required
is exactly the kind of unnecessary attack surface the services lesson
already covered, just at the role level instead of the service level.

```
Get-WindowsFeature | Where-Object {$_.InstallState -eq "Installed"}
Remove-WindowsFeature -Name Web-Server
```

Cross-reference the installed-roles list against what the README says the
server actually needs to provide, the same way you'd cross-reference a
running-services list. Remove what isn't required.

## GPO password enforcement — why `net accounts` isn't the whole story here

The account-and-password-policy lesson's `net accounts` and `secpol.msc`
commands set *local* security policy. On a domain-joined machine, that's
not what actually governs domain accounts — Group Policy, applied down from
a domain controller, wins. Specifically, the **Default Domain Policy** GPO
is where domain-wide password policy is meant to live:

```
Get-GPO -Name "Default Domain Policy"
gpupdate /force
```

`Get-GPO` confirms the policy object exists and its last-modified time;
`gpupdate /force` re-applies Group Policy immediately rather than waiting
for the next automatic refresh cycle, which matters when you need to verify
a change took effect during a time-limited round. The practical takeaway:
if you're on a domain controller or a domain-joined image and password
policy still looks wrong after running `net accounts`, the fix belongs in
the Default Domain Policy (via the Group Policy Management console),
not the local security policy.

## AD/DNS fundamentals

Two pieces matter most at this introductory level:

**Domain Admins is the high-value group.** It's the domain-wide equivalent
of the local Administrators group from the users-and-groups lesson — except
membership grants administrative rights across *every* domain-joined
machine, not just one box.

```
Get-ADGroupMember -Identity "Domain Admins"
```

Audit this list against the README's authorized-administrator information
with even more scrutiny than a local Administrators check, given how much
larger the blast radius of an unauthorized member is.

**DNS is attack surface too.** A domain controller commonly also runs the
domain's DNS server. A poisoned or unauthorized DNS record has the same
practical effect as the hosts-file tampering covered earlier in this
pillar — silently redirecting where a hostname resolves — except it affects
every machine querying that DNS server, not just one.

```
Get-DnsServerResourceRecord -ZoneName "roundzero.local"
```

Reviewing zone records for anything unexplained is the domain-scale version
of the same hosts-file check from the previous lesson.
