Account policy is one of the most reliably scored categories on a Windows
image, and it rewards knowing exactly which tool sets which setting. This
lesson covers length/age/history/lockout — genuinely settable from the
command line — and the one setting that famously isn't: complexity.

## `net accounts` — fast, and mostly complete

`net accounts` is a single command that sets several password-policy values
at once, from an elevated Command Prompt:

```
net accounts /minpwlen:14 /maxpwage:90 /minpwage:1 /uniquepw:5
net accounts
```

The second line with no switches prints the current values — always run it
after a change to confirm.

| Switch | Sets |
|---|---|
| `/minpwlen:N` | Minimum password length |
| `/maxpwage:N` | Maximum password age (days) |
| `/minpwage:N` | Minimum password age (days) |
| `/uniquepw:N` | Password history — how many old passwords are remembered |
| `/lockoutthreshold:N` | Failed logon attempts before lockout |
| `/lockoutduration:N` | How long an account stays locked (minutes) |
| `/lockoutwindow:N` | Time window the failed-attempt counter resets after (minutes) |

If the README gives specific numbers, use those. Otherwise, reasonable
defaults are a minimum length of 14, a max age around 90 days, a min age of
1 day (so the history requirement can't be bypassed by changing the password
five times in one minute), and a history of 5.

## The trap: complexity has no `net accounts` switch

Look at that table again — there is no `/complexity` flag. This is
deliberate on Microsoft's part, and it is the single most common way a team
thinks they've finished account policy when they haven't. `net accounts` sets
length, age, history, and lockout. It does not, and cannot, enable "Password
must meet complexity requirements."

To actually turn complexity on, you need one of:

**GUI (Pro/Enterprise editions):**

```
secpol.msc
```

Navigate to Account Policies → Password Policy → "Password must meet
complexity requirements" → Enabled. In the same screen, set "Store passwords
using reversible encryption" to Disabled.

**Command line (works even without the GUI snap-in):**

```
secedit /export /cfg C:\sec.cfg
:: edit C:\sec.cfg — set PasswordComplexity = 1 and ClearTextPassword = 0
secedit /configure /db %windir%\security\local.sdb /cfg C:\sec.cfg /areas SECURITYPOLICY
```

`secedit /export` dumps the current security policy to a text file you can
edit directly, then `secedit /configure` applies it back. This is the same
underlying mechanism secpol.msc uses — it's just accessible from a script or
a Home-edition box where the snap-in doesn't exist.

## Reversible encryption: why it's always a finding

"Store passwords using reversible encryption" sounds like a minor setting,
but it changes *how the password is stored*, not just how it's checked. A
normal Windows password is stored as a hash — a one-way function that can
verify a login attempt without ever being reversible back to the original
password. Reversible encryption stores the password in a form Windows
*can* decrypt. Anyone who obtains that stored value has the actual password,
not just a hash to crack. This is why it's flagged regardless of how strong
the passwords themselves are — the storage method itself is the
vulnerability. It should always be disabled unless a specific legacy
application on the README genuinely requires it (rare, and worth a second
look if claimed).

## Lockout policy: don't assume "already set" means "correctly set"

Newer Windows builds ship with some lockout policy already enabled by
default. That is not the same as matching the README's numbers or a
reasonable competition baseline — verify the actual values, don't just check
that lockout is "on":

```
net accounts /lockoutthreshold:5 /lockoutduration:30 /lockoutwindow:30
net accounts
```

A threshold of 5 attempts, a 30-minute lockout, and a 30-minute reset window
is a solid default absent other guidance.
