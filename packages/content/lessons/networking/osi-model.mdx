The OSI model gets taught as seven layers to memorize and then mostly
forgotten. That's a waste — it's actually the fastest tool you have for
answering "where is this problem?" when something on a network build isn't
working, which is exactly what the Cisco Challenge tests.

## The seven layers, top to bottom

| # | Layer | What lives here | The one thing to remember |
|---|---|---|---|
| 7 | Application | HTTP, DNS queries, SSH, email | What the user's software actually asked for |
| 6 | Presentation | Encryption, encoding, compression | Data format — is it TLS-wrapped, is it UTF-8 |
| 5 | Session | Sessions/connections between apps | Keeping a conversation alive |
| 4 | Transport | TCP, UDP, port numbers | Reliable delivery (TCP) vs fast-and-loose (UDP) |
| 3 | Network | IP addresses, routers, routing | Getting a packet from one network to another |
| 2 | Data Link | MAC addresses, switches, frames | Getting a frame across one local link |
| 1 | Physical | Cables, signals, ports | Bits on the wire |

A common mnemonic, top to bottom: "All People Seem To Need Data Processing."

## Why a competitor cares about this at all

The Cisco Challenge is a Packet Tracer build plus a knowledge quiz, and both
reward knowing which layer a device or a problem belongs to:

- **Switches operate at layer 2.** They forward frames based on MAC
  addresses and don't understand IP addresses at all.
- **Routers operate at layer 3.** They forward packets between different
  networks based on IP addresses — this is the device that actually
  connects your subnets together.
- **ACLs, ports, and NAT live at layer 3/4** — this is the "device
  configuration" half of the pillar (later lessons).

## The real payoff: narrowing down a problem fast

When something doesn't work, the layers give you a search order instead of
guessing randomly:

1. **Link light dead, cable unplugged?** Layer 1.
2. **Cable's fine, but the device won't get traffic to/from its neighbor on
   the same switch?** Layer 2 — check the VLAN and port mode.
3. **You can reach devices on your own subnet but nothing beyond it?** Layer
   3 — check the IP address, subnet mask, default gateway, and the router's
   routing table.
4. **You can ping the far-end IP just fine, but the actual service (a web
   page, SSH) won't connect?** Layer 4 or above — check whether the port is
   open and the service is actually running.

That's the whole value of the model in a competition setting: it turns "it's
broken" into "it's broken somewhere between layer 3 and layer 4," which cuts
your troubleshooting time dramatically.
