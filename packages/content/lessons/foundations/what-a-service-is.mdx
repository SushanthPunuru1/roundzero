The last two lessons covered what a machine is and who's allowed to touch
it. This one covers the third piece: what's quietly *running* on it right
now, without anyone sitting at the keyboard.

## A service is a program with no one watching it

Most programs you think of run because a person launched them — you double
click a browser, you open a text editor. A **service** (Windows' term) or
**daemon** (Linux's term for the same idea) is different: it starts
automatically, usually when the machine boots, and keeps running in the
background indefinitely, waiting to respond to something. A web server
waits for a browser to connect to it. A print spooler waits for a print
job. An SSH server waits for someone to try to log in remotely.

You can't see most services in a normal desktop view — no window, no icon
in the taskbar. That's exactly why they need a deliberate, separate
review step during hardening: nothing about a machine's visible desktop
tells you what's running underneath it.

## Every running service is something an attacker could try

Here's the idea this whole lesson exists to teach: **a service is a
program an attacker can potentially interact with, without ever touching
your keyboard**. If a service has a bug, or a weak configuration, or
simply doesn't need to exist on this machine, it's a way in — a piece of
what security people call the **attack surface**: the total set of things
available for someone to attempt to exploit.

This gives you a genuinely simple mental model for most of hardening: more
unnecessary services running means more attack surface; fewer means less.
It's why "list every running service and ask whether it needs to be there"
is one of the very first things a hardening checklist asks you to do —
before touching passwords, before touching files, before anything else.

## "Disable what you don't need" — but check what you *do* need first

The natural instinct is to shut down everything unfamiliar. That instinct
is right most of the time and wrong in a way that costs real points: some
services are explicitly required by the scenario (the next lesson's
territory covers this in depth). Stopping or removing a **required**
service is treated as a penalty, not a fix — even though, in isolation,
that service might be exactly the kind of thing you'd otherwise want gone.

So the actual rule has two halves:

1. **Not needed, not required → disable it.** This shrinks attack surface
   for free.
2. **Required by the scenario → keep it running, but configure it as
   securely as it can be.** You can't remove the door, so you make sure
   it locks properly.

Telling those two cases apart is exactly why "read the README first"
(covered later in this domain) comes before "start hardening services."

## The pattern, again

Notice the throughline from the last two lessons: the OS is what runs
everything, accounts and permissions control who's allowed to act, and
services are the *what's currently acting without anyone watching it*
layer. A service review is really just asking, for every single thing
quietly running: does this need to exist here, and if so, is it as locked
down as it can be? That question, asked over and over, is most of what
hardening actually is.
