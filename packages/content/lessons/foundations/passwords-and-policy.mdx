Passwords are the single most common thing this competition asks you to
audit, and most of the mental model people carry into it is wrong in small
but important ways. This lesson fixes that model before you touch a real
password policy setting.

## Your password is never stored as typed

When you set a password, the operating system doesn't keep a list of
plain-text passwords sitting around. Instead, it runs your password
through a **hash function** — a one-way mathematical scramble — and stores
the scrambled result. When you log in later, the OS hashes what you typed
and compares the two scrambled results. If they match, you typed the right
password, even though the system never stored (and can't recover) the
original text.

"One-way" is the important word. A good hash can't be un-scrambled back
into the original password — the only way to find a password that
produces a given hash is to guess passwords and hash each guess, checking
for a match. That's the entire reason password *strength* matters at
all: hashing already protects the stored data from being read directly, so
everything else is about making the guessing process too slow or too
unlikely to succeed.

## Why length beats complexity

Every character added to a password multiplies the number of possible
passwords an attacker has to try before finding yours. A password
requirement that demands "one uppercase, one number, one symbol" in an
8-character password sounds strict, but a genuinely longer password —
even one built from ordinary words — can be dramatically harder to guess,
simply because there are so many more possible combinations at greater
length. This is why modern password guidance leans toward **length
first**, complexity second: `correct horse battery staple` beats `P@ss1!`
in raw guessing difficulty, even though it looks "simpler."

This doesn't mean complexity rules are pointless — they still block the
very weakest passwords (`password1`, `qwerty123`) — just that length is
the bigger lever, and a policy that only enforces complexity while
allowing short passwords is missing the more important control.

## What a password policy actually enforces

A "password policy" is a bundle of separate settings, each catching a
different weak habit:

- **Minimum length** — blocks passwords too short to resist guessing.
- **Complexity** — requires a mix of character types, blocking the most
  trivially guessable passwords.
- **Age / expiration** — how long a password can be used before it must
  change.
- **History / reuse** — how many previous passwords are remembered and
  blocked from being reused.
- **Lockout** — how many wrong login attempts are allowed before an
  account is temporarily locked, which slows down someone trying to guess
  their way in by brute force.

Each of these is a separate, independently-configurable knob — a system
can have a strong minimum length and no lockout at all, or a strict
lockout with no length requirement. Auditing "password policy" during
hardening means checking each of these settings individually, not
assuming one implies the others.

## Why reuse policy still matters even with hashing

A reasonable question: if passwords are already hashed, why does it
matter if someone reuses an old one? The answer is that hashing protects
*this system's storage* — it says nothing about what happens if that same
password was used somewhere else and leaked in an unrelated breach, or was
simply guessed once already. A history policy that blocks cycling back to
recently-used passwords closes that specific gap: even a password that's
already been compromised once, somewhere, can't just come back into
rotation here.

Passwords are one setting; the discipline of checking each part of the
policy, rather than assuming "a password exists" is the same as "the
password policy is fine," is the actual skill this lesson is teaching.
